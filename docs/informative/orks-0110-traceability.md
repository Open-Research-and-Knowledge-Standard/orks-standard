## Compatibility Fixture Matrix

Executable fixtures must cover exact versions in both directions across patch,
minor, major-zero, and major boundaries; absent and present operation rows;
range, nearby, reverse, and transitive inference attempts; structural-invalid
before unsupported; unsupported before resource policy; supported old-version
dispatch; and unchanged input bytes. Unknown versions, families, closed-set
values, required features, critical extensions, known-invalid extensions, and
noncritical opaque preservation must each have separate expected outcomes.

Profile fixtures must prove that Unicode 17.0.0 and `orks-loc:v1`,
accepted JSON, identity framing, every `orks-ref:*:v1` class, and
`orks-markdown:v1` remain immutable across suite versions. A successor
profile uses a distinct identifier and receives no nearby-profile support.

## Migration and Preservation Fixture Matrix

Migration fixtures must cover a same-family successor, same-family ordered
two-parent reconciliation, family change, split, combination, complete
ancestry, non-ancestry provenance, reference dependency,
and evidence closure, mixed-version retained history, unknown historical
versions, new revision and bundle identities,
truthful producer attribution, pending candidate state, separate human review,
stale expected heads, collision, checked limit boundaries, crash injection,
atomic batch visibility, and absence of partial output.

Privacy fixtures must repeat closure, classification, secret scanning, preview,
per-source authorization, and final confirmation for the migrated selection.
They must reject host paths, credentials, prompts, provider output, telemetry,
input-selected routes, implicit fetches, evidence-status laundering, extension
loss, criticality reduction, and side-channel loss.

Every Standard Kernel downgrade fixture must produce refusal without a
candidate or target artifact. Exact-byte preservation fixtures compare the
complete descriptor and entry tree plus bundle identity. Same-version semantic
round trips compare the complete revision semantic value, canonical bytes, and
revision identity. Forward migration fixtures compare the plan-defined
semantic and provenance invariants while proving that new revision identities
do not masquerade as original bytes.

At least two independent implementations from different language ecosystems
must process the same compatibility and migration fixture corpus without
sharing a parser, serializer, migration engine, or expected-output generator.
No implementation output becomes authority; every expected result cites one
task-owned rule and one normative example.
